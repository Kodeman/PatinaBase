The Proposal ThatBecomes the Project

The previous proposal design treated proposals and projects as separate entities connected by a "Create Project" button. That's wrong. In real interior design practice, the proposal IS the project definition. When a client signs, every room, every phase, every line item, every exclusion, and every payment milestone should flow directly into the Opus project system with zero re-entry. This document redesigns the entire flow.

01 — The Problem with v1

Why proposals and projects can't be separate.

The v1 proposal was a beautiful document — vision, concept, selections, investment, timeline, terms. But it treated the proposal as a narrative and the project as a separate operational tool. When the client signed, a "Create Project" button spawned a new entity and someone had to manually re-enter the scope, budget, timeline, and FF&E schedule. That's where scope creep starts, where budgets diverge, and where designers lose hours.

What broke in v1

The proposal said "$24,500 for living and dining rooms" — but the project had no structured scope definition. Which rooms? What's in each room? What's included vs. excluded? How many revision rounds? What happens when the client adds a hallway mid-project?

The FF&E schedule existed in the project but not in the proposal. The proposal had "Product Selections" as a narrative section with product cards. The project had a procurement pipeline with PO numbers. These were two completely different data structures describing the same furniture.

What v2 fixes

The proposal builds the project's data structure as it's written. Room scope, phase timeline, FF&E selections, budget, payment milestones, exclusions, revision limits, and change order process are all structured data underneath the narrative prose. When the client signs, the project doesn't get "created" — it gets "activated."

Every product in the proposal's Selections section is a line item in the FF&E schedule. Every phase in the proposal's Timeline section is a phase in the Opus project. Every dollar in the Investment section maps to a budget row. The proposal and the project are the same object in two views.

Site visit, room scan, intake questionnaire, budget conversation, style assessment. Outputs a project brief. This step is billed separately ($150–500) and can happen before any proposal exists.

Room-by-room definition

Designer defines which rooms, what's in each room (FF&E categories), preliminary product selections vs. allowances, phase structure, fee model, revision limits, and exclusions. This builds the proposal's structural skeleton.

Step 3 · Proposal Document
Narrative wraps the data

The designer writes the vision, concept, and story prose around the structured scope. The client reads a beautiful document. The system stores structured data underneath. Same typography, same page — dual nature.

Step 4 · Activation
Signature → Project

Client signs. The proposal's structured data activates as a live Opus project. Rooms → scope cards. FF&E → procurement pipeline. Phases → timeline. Budget → financials. Payment milestones → invoicing. Zero re-entry.

Change Order Process
Defined upfront in the proposal terms. When a client requests scope changes, this is the process.
How scope changes work

Any modification to rooms, FF&E items, or service phases after proposal signing requires a Scope Change Authorization — a structured form showing: what changed, cost impact (up or down), timeline impact, and client approval signature. Changes are not verbal. The project budget and timeline update only after written authorization. Additional design fees for out-of-scope work are billed at $150/hr.

05 — Scope Change Authorization

When the client adds the hallway.

Scope creep is the number one profitability killer in interior design. The Scope Change Authorization is how Patina handles it — a structured form that captures what's changing, what it costs, what it does to the timeline, and requires a client signature before the work begins. This isn't punitive — it's transparent. Clients appreciate knowing exactly what a change costs before agreeing to it.

What happens when approved

A new "Entryway" room scope card is added to the project. The FF&E schedule adds 6 items at Allowance stage. Budget and timeline update across the project. A new payment milestone may be added. The original signed proposal is annotated with this change order.

What this changes about everything.

Scope Builder creates the skeleton before the narrative. The designer defines rooms, FF&E categories, fee structure, revision limits, and exclusions in a structured interface. Then "Generate Proposal" wraps that data in the editorial typography the client sees. The client reads a beautiful document. The system stores structured, actionable data. Same page, dual nature.

Fixed vs. Allowance vs. TBD replaces the fiction of exact pricing. Real proposals can't price every item — the dining chairs haven't been selected yet. Allowances give the client a budget range ("$300–450/chair × 6") while giving the designer sourcing flexibility. When the product is selected during Design Development, the allowance converts to a fixed line item. The budget math stays honest throughout.

Exclusions are a first-class section, not fine print. "What's not included" matters as much as "what is." Structural work, contractor management, permits, extra rooms — all stated explicitly. When the client asks about the hallway, the designer doesn't have an awkward conversation. She has a Scope Change Authorization form that makes it professional and transparent.

Signing activates, it doesn't create. When the client signs, the proposal doesn't spawn a separate project. The structured data underneath the narrative prose activates as a live Opus project. Rooms become scope cards. FF&E lines enter the procurement pipeline. Phases become the timeline. Budget rows become the financial tracker. Payment milestones become invoicing triggers. The proposal and the project are the same entity in two views.

Change orders reference the original scope. Every scope change shows what the original proposal included, what's being added, the cost impact, and the timeline impact. The client approves digitally. The project updates. The original proposal is annotated. There's never a question about "was this included?" — the answer is in the signed scope document.

Revision limits prevent invisible scope creep. "2 revision rounds per phase" is defined in the proposal, tracked in the project, and enforced with a counter. When the client requests a third round of mood boards, the system alerts the designer: "Revision limit reached. Additional revisions at $150/hr. Send notification to client?" This isn't about being inflexible — it's about making the cost of indecision visible.
