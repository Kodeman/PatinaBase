Interface Design Vision

01 — The Diagnosis

The current portal reads
like software documentation,
not a design tool.

After walking through the existing Designer Portal specification — eight navigation sections, Kanban boards, side panels, wizard modals, tabbed views inside tabbed views — the pattern becomes clear. We've been building a project management tool that happens to live inside a design brand, rather than building a design tool that happens to manage projects.

01

Container Nesting

Cards inside panels inside tabs inside views. Every piece of information is boxed, bordered, and shadowed. The eye has no resting place — it bounces between containers instead of reading the content within them.

02

Navigation Overload

Eight primary destinations, each with three to four sub-sections, plus contextual sidebars and floating action buttons. A designer looking for a client's room scan must navigate through four levels of hierarchy. That's not a tool — it's a filing cabinet.

03

Identity Mismatch

Patina's consumer brand speaks in Playfair Display and warm earth tones. The portal specification reads like a Jira competitor — Kanban columns, bulk actions, sortable tables, badge counts. Two different products under one roof.

04

Widget Overload

The dashboard specification calls for customizable widget grids, revenue charts, pipeline funnels, and performance gauges — all competing for attention simultaneously. When everything demands attention, nothing gets it.

02 — The Philosophy

Let the type do the work.

The best design tools don't look like software. They look like the work itself. A designer's day lives in visual hierarchies — client names, project phases, room dimensions, budget numbers. Present that information with the right typographic weight, at the right size, in the right color, and you don't need a box around it.

01

Type Weight Replaces Containers

A client name in 500-weight Inter at 15px tells you it's important. A project phase in 11px monospaced uppercase tells you it's metadata. A lead score in 36px Playfair tells you it's the headline. No card needed. No shadow. No border. The typographic scale is the interface.

02

Horizontal Rules Replace Tabs

Instead of tabbed navigation that hides information behind clicks, we use the Strata Mark system — thin horizontal lines of decreasing width — to separate content zones. Everything stays on one continuous page. Scroll is better than click when the content is well-structured.

03

The Document Metaphor

A lead detail page shouldn't look like a CRM record. It should read like a design brief — the client's name as a heading, their style preferences as a narrative, the room scan as an embedded image, the budget as a typographic callout. The same information, presented as something a designer would actually want to read.

04

Whitespace as Architecture

In construction, negative space defines the room as much as the walls do. In our interface, generous margins and padding create visual "rooms" where related information lives together — without needing explicit containers to define the boundaries.

05

Progressive Weight, Not Progressive Disclosure

Instead of hiding details behind "view more" toggles, we show everything — but at different typographic weights. Primary information is bold and large. Supporting details are lighter and smaller. Metadata is monospaced and muted. The eye naturally scans what matters.

"The interface should feel like something Leah would pin to her studio wall — not something she'd have to be trained to use."
Design Vision Principle — Patina Designer Portal

08 — Feature Vision

How the key features should feel.

The Lead Inbox

Core Feature

The current specification describes lead cards with badge counts, priority colors, budget indicators, and countdown timers — all visible simultaneously. This creates visual noise that works against the very decision it's trying to support.

Instead, the lead inbox should read like a list of people. Names set in medium-weight Inter. Below each name, a single line of context — project type, location, budget range — in lighter type. To the right, the match score as a large Playfair numeral, because that's the first thing a designer's eye should find.

No colored badges. No urgency indicators fighting for attention. The match score and the response deadline are the only two data points that matter for the initial scan. Everything else reveals itself when a lead is opened into the full brief view.

The Proposal Builder

Core Feature

The current specification calls for a step-by-step wizard with a progress bar — the same pattern used by tax software and airline booking. A designer creating a proposal isn't filling out a form. She's composing a vision.

The proposal builder should feel like a document editor. One continuous page. The design narrative at the top, flowing into mood board imagery, then space plans imported from the 3D viewer, then product selections with inline pricing, and finally terms — all readable top to bottom in a single scroll.

Each section has a Playfair heading and a subtle strata divider. Products are listed by name with their maker story visible, not hidden behind a "details" link. The proposal a client receives should look identical to what the designer sees while creating it — same type, same colors, same layout. No "preview mode" necessary because the edit mode is the preview.

The Room Viewer

Differentiator

This is where the interface steps back entirely. When a designer opens a room scan, the chrome disappears. Full-screen immersion. The 3D model fills the viewport. Navigation controls are minimal — orbit and zoom, period. Measurements overlay on demand.

Product placement happens through a slim inline search at the bottom of the viewport — type a product name, drag it into the space. No side panels. No tabbed product categories. The room is the canvas. Everything else is a tool in your hand, not a menu on a shelf.

When the designer is finished, she can capture a view that embeds directly into the proposal document — same quality, same framing, seamless continuity between the spatial tool and the written brief.

Project Timeline

Simplified

The Kanban board is the default pattern for project management. It's also a pattern that belongs to engineering teams, not design studios. Leah doesn't think in swimlanes. She thinks in phases — consultation, concept, refinement, procurement, installation, walkthrough.

The project view should be a simple vertical timeline. Each phase is a heading. Under it, the tasks and milestones for that phase, with their dates and status shown through type weight — bold for active, regular for upcoming, light for complete. A thin progress line runs vertically alongside, showing where the project stands.

Click any phase heading and it expands inline to show the full detail — no modal, no page change, no loss of context. The designer stays oriented within the larger project arc while working on the details.

09 — Interaction Patterns

Invisible until needed.

Actions should be contextual, not omnipresent. A button that appears when you hover over a lead name is better than a permanent action bar competing for attention with the content it serves.

Hover to Reveal

Primary actions appear on hover. A lead item shows "Accept" and "Pass" only when the cursor is over it. On mobile, these surface with a long-press or swipe gesture.

Desktop
Hover → Actions fade in at right edge
Mobile
Swipe right → Accept / Swipe left → Pass

Inline Expansion

Detail views expand inline rather than navigating to new pages. Clicking a project in the timeline expands its tasks below the heading, pushing other content down. Context is never lost.

Transition
300ms ease with content-aware resize
Collapse

Typographic Transitions

State changes happen through type, not color fills. An accepted lead's name shifts from regular to medium weight. A completed phase shifts from charcoal to muted gray. Subtle, informative, brand-consistent.

10 — Implementation Path

How we build this.

This isn't a redesign of every screen simultaneously. It's a phased approach that starts with the two views Leah uses most — the dashboard and the lead brief — and expands outward.

01

Foundation Sprint — The Type System

Build the typographic design tokens: Playfair Display at three weights, Inter at three weights, DM Mono for metadata. Define the complete scale — sizes, line-heights, letter-spacing, and color assignments. This becomes the shared language every screen speaks. Implement as CSS custom properties in the Patina Design System.

02

Core Screens — Dashboard & Lead Brief

Implement the dashboard as shown in this vision — typographic greeting, inline metrics, lead list, and project timeline. Then the lead brief view with its document-style layout. These two screens handle 80% of daily workflow. Ship them, test with Leah, iterate.

03

Extended Views — Projects & Proposals

Apply the typographic system to project management (vertical timeline) and the proposal builder (document editor). These build on the patterns established in the first sprint, so the visual language is already proven.

04

Studio Zone — Earnings & Portfolio

The business management features — earnings dashboard, commission tracking, portfolio showcase. These are lower-frequency screens but still need to speak the same typographic language. Playfair numerals for financial data. Mono labels for date ranges. Clay accents for positive trends.

This is how it should feel.

You're reading this document right now in the same typographic system proposed for the Designer Portal. The page titles in Playfair. The body copy in Inter. The metadata in monospace. No containers-in-containers. No tabs. No cards with drop shadows competing for your eye.

If this document feels clear, readable, and distinctly like Patina — then the Designer Portal can feel the same way. The type is doing the work. The whitespace is building the rooms. And the interface is getting out of the way of the content.

That's the vision. Let's build it.
